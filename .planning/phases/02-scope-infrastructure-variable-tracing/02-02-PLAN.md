### SCOPE-04: Lambda/higher-order function context

#### $map -- element binding
- `extractPaths('$map(items, function($v) { $v.name })')` -> `[{ path: "items" }, { path: "items.name" }]`
  $v is bound to the element of "items". $v.name resolves to "items.name". "items" is read as the first argument.
- `extractPaths('$map(orders.items, function($v, $i) { $v.price })')` -> `[{ path: "orders.items" }, { path: "orders.items.price" }]`
  $v bound to element of "orders.items", $i is index (non-data). $v.price -> "orders.items.price".
- `extractPaths('$map(data, function($v, $i, $a) { $v.x + $a })')` -> result should include `[{ path: "data" }, { path: "data.x" }]`
  $v bound to element of "data", $i=index (ignored), $a=full array bound to "data". $v.x -> "data.x", $a -> "data".

#### $filter -- element binding
- `extractPaths('$filter(orders, function($v) { $v.total > 100 })')` -> `[{ path: "orders" }, { path: "orders.total" }]`
  $v bound to element of "orders". $v.total -> "orders.total".

#### $reduce -- accumulator + element
- `extractPaths('$reduce(values, function($prev, $curr) { $prev + $curr })')` -> `[{ path: "values" }]`
  $prev is accumulator (bound to "values" paths), $curr is element (bound to "values" paths). Both resolve to "values". Deduped: just "values". Actually: $prev + $curr within the function body -- $prev resolves to "values", $curr resolves to "values", so body paths are ["values", "values"]. Plus the argument "values". Deduped: ["values"].

#### $each -- value/key
- `extractPaths('$each(data, function($v, $k) { $v })')` -> `[{ path: "data" }]`
  $v bound to value of "data" entries, $k is key (non-data). $v -> "data".

#### $sift -- value/key
- `extractPaths('$sift(record, function($v, $k) { $v > 10 })')` -> `[{ path: "record" }]`
  $v bound to value of "record", $k is key (non-data).

#### Lambda closure capture
- `extractPaths('($prefix := a.b; $map(items, function($v) { $prefix }))')` -> `[{ path: "a.b" }, { path: "items" }]`
  Lambda captures $prefix from enclosing scope. $prefix resolves to ["a.b"]. "items" from first arg. "a.b" from both the bind RHS and the closure. Deduped.

#### Lambda parameter shadowing built-in
- `extractPaths('$map(items, function($sum) { $sum.x })')` -> `[{ path: "items" }, { path: "items.x" }]`
  Lambda parameter $sum shadows the built-in $sum. $sum resolves to element of "items", so $sum.x -> "items.x".

### SCOPE-04: Apply operator (~>)
- `extractPaths('items ~> $map(function($v) { $v.name })')` -> `[{ path: "items" }, { path: "items.name" }]`
  Apply: lhs "items" becomes first argument to $map. Equivalent to $map(items, function($v) { $v.name }).
- `extractPaths('data ~> $filter(function($v) { $v.active })')` -> `[{ path: "data" }, { path: "data.active" }]`
  Apply with $filter.
- `extractPaths('items ~> $sum()')` -> `[{ path: "items" }]`
  Apply with non-higher-order built-in: items becomes first arg to $sum. Pass-through: "items" extracted.

### SCOPE-05: Custom function calls
- `extractPaths('($fn := function($x) { $x.name }; $fn(account))')` -> `[{ path: "account" }, { path: "account.name" }]`
  Custom function: $fn is bound to a lambda. When $fn(account) is called, $x is bound to ["account"] (the argument paths), then body $x.name resolves to "account.name". "account" from the call-site argument.
- `extractPaths('($fn := function($a, $b) { $a.x + $b.y }; $fn(data1, data2))')` -> `[{ path: "data1" }, { path: "data2" }, { path: "data1.x" }, { path: "data2.y" }]`
  Multi-param custom function.
- `extractPaths('$unknownFunc(a.b, c.d)')` -> `[{ path: "a.b" }, { path: "c.d" }]`
  Unknown function (not in scope, not built-in): pass-through argument extraction.

### Edge cases
- `extractPaths('$map(items, function($v) { $map($v.children, function($c) { $c.name }) })')` -> `[{ path: "items" }, { path: "items.children" }, { path: "items.children.name" }]`
  Nested higher-order: outer $v = element of "items", inner $c = element of $v.children = "items.children". $c.name -> "items.children.name".

### Step 1: Enhance the function handler in src/walker.ts

Replace the pass-through function handler with lambda-aware logic:

```typescript
function walkFunction(node: FunctionNode, scope: ScopeTracker): string[] {
  const funcName = node.procedure.value;
  const args = node.arguments;
  const paths: string[] = [];

  // Step 1: Extract paths from all non-lambda arguments (they're data reads)
  for (const arg of args) {
    if (arg.type !== "lambda") {
      paths.push(...walkNode(arg, scope));
    }
  }

  // Step 2: Check if this is a call to a known higher-order function
  const semantics = HIGHER_ORDER_SEMANTICS[funcName];
  if (semantics) {
    // Find the lambda argument
    const lambdaArg = args.find(a => a.type === "lambda") as LambdaNode | undefined;
    if (lambdaArg) {
      // Get the data argument (first non-lambda arg) paths for binding
      const dataArg = args.find(a => a.type !== "lambda");
      const dataArgPaths = dataArg ? walkNode(dataArg, scope) : [];

      // Walk lambda body with parameter bindings
      paths.push(...walkLambdaWithBindings(lambdaArg, dataArgPaths, semantics, scope));
    }
    return paths;
  }

  // Step 3: Check if this is a call to a custom function (lambda in scope)
  const resolvedFunc = resolveVariable(scope, funcName);
  // If the function name resolves to something, this is a custom function call.
  // But we need the actual lambda node, not paths. So we need a different approach.
  // Instead: look up if funcName is bound to a lambda in scope via a separate lookup.
  // For custom functions, we need to store the lambda node itself, not just paths.
  // DECISION: For SCOPE-05, we'll check if the procedure is a lambda node directly
  // passed as an argument, OR if it's a variable that was bound via $fn := function(...).
  // For the latter, we need scope to store lambda references.
  // APPROACH: Store lambda metadata alongside paths in scope. When a bind's RHS is
  // a lambda, store a special marker. When calling, check for the marker and walk
  // the lambda body with argument bindings.
  //
  // SIMPLER APPROACH: If the function call's procedure resolves to nothing in scope
  // and it's not a built-in, just pass-through all arguments (already done above).
  // If it IS in scope (custom function), we need the lambda.
  // For now: pass-through all lambda arguments too for non-higher-order functions.
  // The lambda body paths are extracted when the lambda was first defined (bind RHS).

  // Step 3a: For non-higher-order built-ins (like $sum, $count) - already handled above
  // Step 3b: For unknown functions - already handled above (pass-through)

  // Step 4: If there's a lambda argument and this is NOT a higher-order built-in,
  // still walk the lambda body with the current scope (closure)
  for (const arg of args) {
    if (arg.type === "lambda") {
      const lambdaNode = arg as LambdaNode;
      const lambdaScope = childScope(scope);
      paths.push(...walkNode(lambdaNode.body, lambdaScope));
    }
  }

  return paths;
}
```

**Key helper -- walkLambdaWithBindings:**
```typescript
function walkLambdaWithBindings(
  lambda: LambdaNode,
  dataArgPaths: string[],
  semantics: Record<number, string>,
  parentScope: ScopeTracker,
): string[] {
  let lambdaScope = childScope(parentScope);

  // Bind each lambda parameter based on its semantic role
  for (let i = 0; i < lambda.arguments.length; i++) {
    const param = lambda.arguments[i];
    const role = semantics[i];

    if (!role) continue; // more params than semantics knows about

    switch (role) {
      case "element":
      case "value":
      case "left":
      case "right":
      case "curr":
        // Bound to element of the data argument
        lambdaScope = bindVariable(lambdaScope, param.value, dataArgPaths);
        break;
      case "index":
      case "key":
        // Non-data-path: bind to empty (produces no paths when referenced)
        lambdaScope = bindVariable(lambdaScope, param.value, []);
        break;
      case "array":
      case "accumulator":
      case "prev":
        // Bound to the full collection/accumulator
        lambdaScope = bindVariable(lambdaScope, param.value, dataArgPaths);
        break;
    }
  }

  return walkNode(lambda.body, lambdaScope);
}
```

### Step 2: Add apply operator handler

```typescript
case "apply": {
  const applyNode = node as ApplyNode;
  const paths: string[] = [];

  // Extract paths from the lhs (it's a data read)
  const lhsPaths = walkNode(applyNode.lhs, scope);
  paths.push(...lhsPaths);

  // The rhs is typically a FunctionNode. Prepend lhs as first argument.
  if (applyNode.rhs.type === "function") {
    const funcNode = applyNode.rhs as FunctionNode;
    // Create a synthetic function node with lhs prepended to arguments
    const augmentedFunc: FunctionNode = {
      ...funcNode,
      arguments: [applyNode.lhs, ...funcNode.arguments],
    };
    // Walk the augmented function (without re-extracting lhs paths since we already did)
    // Actually, walkFunction will re-walk the first argument, which is fine -- dedup handles it
    paths.push(...walkFunction(augmentedFunc, scope));
  } else {
    // Unusual: rhs is not a function node. Walk it normally.
    paths.push(...walkNode(applyNode.rhs, scope));
  }

  return paths;
}
```

### Step 3: Add custom function tracing (SCOPE-05)

For custom function calls where the procedure resolves to a lambda in scope, we need to trace arguments into the lambda body. This requires storing lambda node references in scope alongside paths.

**Approach:** Extend the ScopeTracker to optionally store lambda references. When `$fn := function($x) { $x.name }` is processed:
1. The bind handler walks the lambda's body with current scope (closure capture) to extract paths -- but actually for a lambda definition, no paths are read until it's called.
2. Store the lambda node reference in scope alongside the (empty) paths.
3. When `$fn(account)` is called, look up $fn in scope, find the lambda, bind call arguments to lambda parameters, walk body.

**Implementation approach for custom functions:**

Add an optional `lambdas` map to ScopeTracker (or use a parallel storage mechanism). The simplest approach: in `walkFunction`, when the procedure name resolves in scope AND the original bind RHS was a lambda, treat it as a custom function call.

Alternative simpler approach: In the bind handler, when the RHS is a `lambda` node, store a reference. The scope module can carry an additional `lambdas: Map<string, LambdaNode>` alongside `bindings`.

**Decision:** Extend ScopeTracker with an optional `lambdas` field. This keeps the immutable pattern but adds lambda storage. Update `bindVariable` and add `bindLambda`/`resolveLambda` functions.

If this is too complex for this plan's scope, a viable fallback: when `$fn(arg)` is called and $fn is not a built-in, walk the arguments as pass-through (already done in Plan 01). Custom function body tracing only fires if the function procedure is an inline lambda (not a variable reference). This covers most real-world cases but misses the `$fn := function(...); $fn(...)` pattern.

**Preferred:** Full implementation with lambda storage in scope. Minor additions to src/scope.ts and src/walker.ts.
