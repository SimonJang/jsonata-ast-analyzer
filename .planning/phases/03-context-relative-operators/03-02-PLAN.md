### EXPR-07: Sort expression extraction

#### Basic sort
- `extractPaths('items^(price)')` -> `[{ path: "items" }, { path: "items.price" }]`
  Sort step at index 1 in PathNode.steps. Context prefix from steps[0..0] = `items`. Walk sort term expression `price` -> `["price"]`. Prefix: `items.price`. Base path: `items` (buildPathString skips sort step).

#### Multi-key sort
- `extractPaths('items^(>price, <date)')` -> result contains `items`, `items.price`, `items.date`
  Two sort terms. Both term expressions are walked and prefixed with `items`. Direction flags (descending: true/false) are ignored.

#### Sort with path continuation
- `extractPaths('items^(price).name')` -> result contains `items.name` and `items.price`
  Steps: `[items, sort, name]`. buildPathString produces `items.name` (sort skipped). Sort context prefix from steps[0..0] = `items`. Sort key `price` -> `items.price`.

#### Multi-step path with sort
- `extractPaths('account.items^(price)')` -> result contains `account.items` and `account.items.price`
  Steps: `[account, items, sort]`. Sort is at index 2. Context prefix from steps[0..1] = `account.items`. Sort key `price` -> `account.items.price`. Base path: `account.items`.

#### Sort with complex sort key expression
- `extractPaths('items^(price * quantity)')` -> result contains `items`, `items.price`, `items.quantity`
  Sort key is a binary expression. Walk it: `["price", "quantity"]`. Prefix each: `items.price`, `items.quantity`.

### EXPR-08: Transform operator

#### Basic transform
- `extractPaths('| Account | {"name": FirstName} |')` -> result contains `Account` and `Account.FirstName`
  Pattern is `Account` (path). Update is a unary `{` node with key `"name"` (string, not extracted) and value `FirstName` (path). Walk update -> `["FirstName"]`. Prefix with pattern path `Account` -> `Account.FirstName`.

#### Transform with delete clause
- `extractPaths('| Account | {"name": FirstName}, ["oldField"] |')` -> result contains `Account` and `Account.FirstName` but NOT `oldField`
  Delete clause contains string literals (field names to remove). Per decision: no paths extracted from delete clause.

#### Transform with multi-step pattern
- `extractPaths('| data.Account | {"name": FirstName} |')` -> result contains `data.Account` and `data.Account.FirstName`
  Pattern is `data.Account`. Update value `FirstName` prefixed with full pattern path.

#### Transform with multiple update values
- `extractPaths('| Account | {"first": FirstName, "last": LastName} |')` -> result contains `Account`, `Account.FirstName`, `Account.LastName`
  Multiple key-value pairs in the update object constructor. Each value gets prefixed.

### Group-by (supplementary to EXPR-07 -- sort & group share the context-relative prefix pattern)

#### Basic group-by
- `extractPaths('items{category: price}')` -> result contains `items`, `items.category`, `items.price`
  Group-by is on PathNode.group. Key expression `category` and value expression `price` are both context-prefixed with the base path `items`.

#### Group-by with multi-step path
- `extractPaths('account.items{category: price}')` -> result contains `account.items`, `account.items.category`, `account.items.price`
  Base path is `account.items`. Both key and value expressions prefixed with it.

### Step 2: Add sort step handling to walkPath in src/walker.ts

In the step iteration loop (added in Plan 01 for filter stages), add sort step detection:

```typescript
// In the step iteration loop within walkPath:
if (step.type === "sort") {
  const sortNode = step as SortNode;
  // Context prefix: steps BEFORE the sort step (NOT including it)
  const contextPrefix = buildPathString(node.steps.slice(0, i)) ?? "";
  for (const term of sortNode.terms) {
    const termPaths = walkNode(term.expression, scope);
    paths.push(...prefixPaths(contextPrefix, termPaths));
  }
  // Don't add sort step to path segments (buildPathString already skips it)
}
```

**CRITICAL: Off-by-one avoidance.** Sort prefix uses `steps[0..i-1]` (i.e., `slice(0, i)`), NOT `slice(0, i+1)`. The sort step itself is NOT a path segment -- the collection being sorted is described by the preceding steps. Contrast with filter: filter prefix uses `steps[0..stepIndex]` (inclusive) because the filter is a stage ON the name step that IS part of the path.

### Step 4: Add transform node handler to walkNode

Add a new case to the walkNode switch:

case "transform":

Implement the helper:

```typescript
/**
 * Handle transform operator: | pattern | update, delete |
 * Pattern is walked for base paths.
 * Update values are walked via walkNode (reusing walkUnary for "{" nodes)
 * and prefixed with the pattern path.
 * Delete clause contains string literals only -- no paths extracted.
 */
function walkTransform(node: TransformNode, scope: ScopeTracker): string[] {
  const paths: string[] = [];

  // Walk pattern for base paths
  const patternPaths = walkNode(node.pattern, scope);
  paths.push(...patternPaths);

  // Walk update and prefix results with pattern path
  if (node.update) {
    const patternPrefix = patternPaths.length > 0 ? patternPaths[0] : "";
    const updatePaths = walkNode(node.update, scope);
    paths.push(...prefixPaths(patternPrefix, updatePaths));
  }

  // Delete clause: string literals only, no paths extracted
  // (intentionally not walked)

  return paths;
}
```

**Key insight for transform update:** The update node is a unary `{` (object constructor). When `walkNode` dispatches on it, it hits the existing `walkUnary` handler (case `{`), which walks values but NOT keys. This is exactly correct: update keys define output field names (not data reads), while update values are data reads that should be prefixed with the pattern path.
