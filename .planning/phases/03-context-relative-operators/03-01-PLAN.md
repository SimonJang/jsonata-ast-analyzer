## RED Phase -- Tests to write FIRST (all new tests must fail initially)

### EXPR-03: Filter predicate path extraction

#### Basic filter predicate
- `extractPaths('items[price > 10]')` -> `[{ path: "items" }, { path: "items.price" }]`
  The filter predicate `price > 10` is walked, producing `["price"]`. This is prefixed with the collection path `items`, yielding `items.price`. The base path `items` is also extracted.

#### Filter with comparison to literal
- `extractPaths('items[price > 10]')` -> includes `items.price` but NOT a path for `10` (it's a literal)

#### Filter in middle of multi-step path
- `extractPaths('account.orders[total > 100].items')` -> `[{ path: "account.orders.items" }, { path: "account.orders.total" }]`
  The filter is on `orders` at step index 1. Context prefix = `account.orders` (steps[0..1]). Filter walks `total > 100`, gets `["total"]`, prefixes with `account.orders` -> `account.orders.total`. Base path is `account.orders.items` (buildPathString skips filter stages).

#### Boolean coercion filter (bare name)
- `extractPaths('items[active]')` -> `[{ path: "items" }, { path: "items.active" }]`
  Bare name `active` in filter position gets context-prefixed: `items.active`.

#### Nested filters
- `extractPaths('orders[items[price > 10]]')` -> result contains `orders`, `orders.items`, `orders.items.price`
  Inner filter: walk `price > 10` -> `["price"]` -> prefix with `items` -> `items.price`. Inner path result: `["items", "items.price"]`. Outer filter: prefix with `orders` -> `["orders.items", "orders.items.price"]`. Plus base `orders`.

#### Filter with external variable reference
- `extractPaths('($threshold := 50; items[price > $threshold])')` -> result contains `items` and `items.price` but NOT `items.50` or similar -- variable-resolved paths should NOT be context-prefixed
  NOTE: Given the prefix-after-walk approach, all paths returned by walking the filter expression get prefixed. Since `$threshold` resolves to `[]` (it was bound to a literal `50` which produces no paths), there is nothing from the variable to incorrectly prefix. This test primarily verifies that the literal binding produces no leakage.

#### Focus variable binding
- `extractPaths('items@$v[type = "A"]')` -> result contains `items` and `items.type`
  When `@$v` is present on a name step, `$v` is bound to the context path `["items"]` in a child scope before walking the filter expression. The filter `type = "A"` walks to produce `["type"]`, which gets prefixed with `items`.

### EXPR-06: Array index vs filter distinction

#### Numeric index (positive)
- `extractPaths('items[0]')` -> `[{ path: "items" }]`
  `filter.expr.type === "number"` -> treat as array indexing, no path extraction from the filter.

#### Numeric index (negative)
- `extractPaths('items[-1]')` -> `[{ path: "items" }]`
  Negative numeric literal also recognized as array indexing.

#### Variable as index (over-approximate as filter)
- `extractPaths('($i := 0; items[$i])')` -> result contains `items`
  `$i` is a variable reference (type "variable"), not a number literal. Per decision: non-numeric filter expressions are treated as filter predicates (over-approximate). `$i` resolves to `[]` (bound to literal 0), so no additional paths beyond `items`.

#### Multiple stages on one step
- `extractPaths('items[price > 10][0]')` -> result contains `items` and `items.price`
  First stage is a filter predicate (walks `price > 10`). Second stage is a numeric index (skipped). Both are stages on the same `items` name step.

### Step 2: Add prefixPaths utility to src/walker.ts

Create a shared utility that prefixes an array of paths with a context string:

```typescript
/**
 * Prefix each path with a context string.
 * Used by context-relative operators (filter, sort, group-by, transform).
 * Empty prefix or empty paths are handled gracefully.
 */
function prefixPaths(prefix: string, paths: string[]): string[] {
  if (!prefix) return paths;
  return paths.map(p => p ? `${prefix}.${p}` : prefix);
}
```

### Step 3: Modify walkPath in src/walker.ts to handle filter stages

The current `walkPath` builds a single path from all steps. It needs to be enhanced to:

1. Iterate through each step in the path
2. For each name step, check if it has `stages`
3. For each filter stage: check if numeric (skip) or predicate (walk + prefix)
4. Compute the context prefix from steps[0..currentStepIndex] (inclusive)
5. Handle focus variable binding (`@$v`) by creating a child scope

Key implementation detail for **Pitfall 4 (variable path discrimination)**: The prefix-after-walk approach as locked by user decisions means we walk the filter expression normally and prefix ALL resulting paths. Since variables resolve through the scope chain to absolute paths, and the user decision says "external variable references resolve normally through scope chain -- no context prefix applied to variable-resolved paths," we need to handle this.

**Practical approach:** Walk the filter expression in a child scope. Variable references resolve to their bound paths (already absolute). Direct field accesses return relative paths. The prefixing should only apply to relative paths from the filter expression, not to variable-resolved absolute paths.

**Simplest correct approach:** Since the walker already resolves variables to absolute paths via `walkVariable`, and direct field references return relative paths via `walkPath`/name returns, use a **two-pass strategy within walkPath's stage handling**:
- Create a child scope for the filter context
- If a focus variable exists (`step.focus`), bind it to the context prefix paths
- Walk the filter expression with the child scope
- The results will be a mix of absolute (variable-resolved) and relative (field access) paths
- Since we can't easily distinguish them from the flat string array, apply prefix-after-walk to all results

**IMPORTANT NOTE:** In practice, this means variable-resolved paths DO get prefixed, which is technically over-approximation. For the common case where variables resolve to empty (`$threshold := 50` -> `[]`) or to paths from outside the filter context, this produces a superset. Per the project's over-approximation principle ("report a superset of actual paths rather than risk missing paths"), this is acceptable. If a variable resolves to e.g. `["external.field"]`, the prefix would produce `items.external.field` which is over-approximate but safe.

**Focus variable binding handles the common case correctly:** When `@$v` is used, `$v` is bound to the context path. `$v.name` in the filter resolves via scope to `items.name` -- no prefixing needed because variable resolution already produces the absolute path. The prefixing then only affects bare field accesses in the filter.

```typescript
function walkPath(node: PathNode, scope: ScopeTracker): string[] {
  // Handle variable steps (existing logic preserved)
  const varStepIndex = node.steps.findIndex((s) => s.type === "variable");
  if (varStepIndex >= 0) {
    // ... existing variable step handling (unchanged) ...
  }

  // NEW: Iterate steps and handle stages (filter/sort)
  const paths: string[] = [];

  for (let i = 0; i < node.steps.length; i++) {
    const step = node.steps[i];

    // Handle filter stages on name steps
    if (step.type === "name") {
      const nameStep = step as NameNode;
      if (nameStep.stages && nameStep.stages.length > 0) {
        // Compute context prefix: steps[0..i] inclusive
        const prefixSteps = node.steps.slice(0, i + 1);
        const contextPrefix = buildPathString(prefixSteps) ?? "";

        // Create child scope for filter context
        let filterScope = childScope(scope);

        // Bind focus variable if present (@$v)
        if (nameStep.focus) {
          filterScope = bindVariable(filterScope, nameStep.focus, contextPrefix ? [contextPrefix] : []);
        }

        for (const stage of nameStep.stages) {
          if ((stage as any).type === "filter") {
            const filterStage = stage as FilterStage;
            // EXPR-06: Numeric index guard
            if (filterStage.expr.type === "number") {
              continue; // Array indexing -- no paths extracted
            }
            // Filter predicate -- walk and prefix
            const filterPaths = walkNode(filterStage.expr, filterScope);
            paths.push(...prefixPaths(contextPrefix, filterPaths));
          }
        }
      }
    }

    // Sort step handling deferred to Plan 02
  }

  // Build the base path (existing behavior)
  const basePath = buildPathString(node.steps);
  if (basePath) {
    paths.push(basePath);
  }

  return paths;
}
```
